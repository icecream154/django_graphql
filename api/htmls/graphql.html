GraphQL

GraphQL是一个新的API标准，它提供了一个比REST(具象状态传输)更有效、更强大和更灵活的替代方案。 它是由Facebook开发和开源的，现在由来自世界各地的大型公司和个人社区维护。 在其核心上，GraphQL支持声明性数据获取，客户可以指定它需要从API获取什么数据。 与返回固定数据结构的多个端点不同，GraphQL服务器只公开单个端点，并使用客户机请求的精确数据进行响应。

在过去的几年中，API领域发生了根本性的变化，特别是有三个因素一直在挑战api的设计方式。 不断增长的手机使用量、低功耗设备和不稳定的网络是Facebook开发GraphQL的最初原因。 GraphQL最小化了需要通过网络传输的数据量，从而极大地改善了在这些条件下运行的应用程序。 运行客户端应用程序的前端框架和平台的异构环境使得构建和维护一个能够满足所有不同客户端的需求的API 变得非常困难，而使用GraphQL，每个客户端都可以精确地访问它需要的数据。

同时持续部署已经成为许多公司的标准，快速迭代和频繁的产品更新是必不可少的。 对于REST api，通常需要修改服务器公开数据的方式，以满足客户端的特定需求和设计更改。 这阻碍了快速开发实践和产品迭代。

使用 GraphQL 首先消除了服务器返回冗余数据的场景，其次对于一些嵌套查询， 如果有 n 个列表项，每一项都需要发送一个单独的查询从服务器中获取详细的信息， 那么就会出现请求 n+1 的问题。单个页面需要发送的请求数量与列表长度相关联， 这不是一个明智的设计方式。而多亏了GraphQL的灵活性，客户端使用单个请求解决了问题， 同时客户端上的更改不需要在服务器上做任何额外的工作。这使得反馈周期和产品迭代更快。

当每个客户端精确指定它感兴趣的信息时，GraphQL允许对后端请求的数据有细粒度的洞察， 可以深入了解可用数据是如何被使用的。例如，这可以帮助开发API，并废弃不再被任何客户 端请求的特定字段。使用GraphQL，您还可以对服务器处理的请求进行低级性能监视。 GraphQL使用所谓的解析器函数的概念来收集客户机请求的数据。检测和测量这些解析器的性能 可以提供关于系统瓶颈的重要见解。GraphQL的另一个主要好处是它的类型系统。 GraphQL使用强类型系统来定义API的功能。API中公开的所有类型都是使用GraphQL模式 定义语言在模式中编写的。此模式还充当客户机和服务器之间的契约，以定义客户机如何访问数据。 一旦定义了模式，在前端和后端工作的团队就可以在没有进一步通信的情况下完成他们的工作， 因为它们都知道通过网络发送的数据的明确结构。前端团队可以通过模拟所需的数据结构来轻松 地测试他们的应用程序。一旦服务器准备好了，客户端应用程序就可以从实际的API加载数据。
